# コード品質検証

**目的:** 実装コードの品質を5つの観点から検証し、要件/設計との整合性を確認します。

**引数:**
- `/validate-code [パス]` - 指定したパス（ファイルまたはディレクトリ）を検証
- `/validate-code [.steeringパス]` - 指定したsteeringディレクトリの実装を検証
- `/validate-code` - 最新のsteeringディレクトリを自動検出して検証
- `/validate-code --from-report [フォルダ名]` - 既存の検証レポートの指摘事項に対する再検証
- `/validate-code --from-report [ファイルパス]` - 指定したvalidation-report.mdに対する再検証

**`--from-report`の引数解釈:**

| 入力 | 解釈 |
|------|------|
| `--from-report 20260215-x-trend-v2` | `.steering/20260215-x-trend-v2/validation-report.md` |
| `--from-report .steering/.../validation-report.md` | 指定ファイルそのもの |

> `--from-report`モードでは通常の5観点フル検証をスキップし、元レポートの指摘事項に焦点を絞った再検証を行います。

---

## モード分岐

引数に`--from-report`が含まれる場合は **[再検証モード](#再検証モードfrom-report)** へ進む。
それ以外の場合は **[ステップ1: 検証対象の特定](#ステップ1-検証対象の特定)** へ進む。

---

## 検証観点

| # | 観点 | 内容 |
|---|------|------|
| 1 | スペック整合性 | 要件/設計との対応確認 |
| 2 | コード品質 | コーディング規約、ベストプラクティス |
| 3 | テストカバレッジ | テストの網羅性と品質 |
| 4 | セキュリティ | 脆弱性、機密情報の取り扱い |
| 5 | パフォーマンス | 効率性、リソース使用 |

---

## 実行ステップ（概要）

検証は以下の8ステップで進めます。各ステップは共通ロジック `Skill('validation')` の対応モードを使用します。**各ステップの詳細な手順・コマンド例・出力フォーマットは [reference.md](./reference.md) を参照してください。検証を実行する前に必ず reference.md を読むこと。**

| ステップ | 内容 | 使用モード |
|----------|------|------------|
| 1 | 検証対象の特定（引数解析・関連ファイル収集） | - |
| 2 | スペックとの整合性検証 | validation: スペック整合性モード |
| 3 | コード品質検証（ruff/mypy/ベストプラクティス） | validation: コード品質モード |
| 4 | テストカバレッジ検証 | validation: テストカバレッジモード |
| 5 | セキュリティ検証 | validation: セキュリティモード |
| 6 | パフォーマンス検証 | validation: パフォーマンスモード |
| 7 | 検証レポートの生成 | validation: レポート生成モード |
| 8 | レポートの保存（steeringディレクトリへ） | - |

---

## 判定基準

### 各項目の合否

| 項目 | PASS条件 | CONDITIONAL_PASS条件 |
|------|----------|----------------------|
| スペック整合性 | 100%充足 | 90%以上充足 |
| コード品質 | エラー0件 | 軽微な警告10件以下 |
| テストカバレッジ | 80%以上 | 60%以上 |
| セキュリティ | 高0件、中0件 | 高0件、中3件以下 |
| パフォーマンス | 高0件 | 高0件、中3件以下 |

### 総合判定

| 条件 | 判定 |
|------|------|
| 全項目PASS | **PASS** |
| 高優先度の問題が1件以上 | **FAIL** |
| CONDITIONAL_PASSが1項目以上 | **CONDITIONAL_PASS** |

### FAIL時の対応

1. 問題箇所を特定
2. 修正コードを提案
3. `/validate-code`を再実行して確認

---

## 注意事項

- 検証は破壊的な変更を行いません（読み取りのみ）
- 大規模なコードベースでは対象を絞って実行することを推奨
- テストが存在しない場合は「テストカバレッジ」はスキップ可能

## 再検証モード（--from-report）

`--from-report` 指定時は通常の5観点フル検証をスキップし、元レポート（validation-report.md）の指摘事項に焦点を絞った再検証を行います。手順（FR-1〜FR-6）・出力フォーマット・判定基準は **[reference.md](./reference.md)** を参照してください。
